# 《一人称》双语对照与注音对接规范

## 1. 目标

翻译完成后的主交付形态，不应是“纯原文版”或“纯译文版”，而应是：

**以阅读单元为边界、以段落为最小对照单位的双语对照版。**

这个规范的目标有三点：

- 让读者可以逐段对照原文与译文
- 让需要注音或释义的词能稳定落到前端渲染层
- 让现有前后端项目只需要做渲染和数据消费，不需要重新理解文本结构

---

## 2. 最终展现原则

### 2.1 对照粒度

默认对照粒度不是整章，也不是整封信，而是：

- 一级边界：`第 N 组通信`
- 二级边界：`阅读单元`
- 三级边界：`段落`

也就是说，最终前端应至少能做到“逐段对照”。

不建议只做以下两种形式：

- 整组原文在左、整组译文在右
- 全文原文在前、全文译文在后

因为这会直接削弱：

- 书信体中细小语气差异的可见性
- 诗与正文之间的照应
- 补遗和异文的结构辨识度

### 2.2 对照单位必须保留结构身份

每个对照块至少要保留以下信息：

- `group_id`
- `unit_id`
- `para_id`
- `speaker`
- `type`
- `source_md`
- `target_md`
- `term_notes`

其中：

- `group_id` 对应 `第01组` 这类通信组
- `unit_id` 对应该组内的阅读单元
- `para_id` 对应单元内的段落序号
- `speaker` 取值至少包括：`我`、`老师`、`未判定`
- `type` 取值至少包括：`来信`、`回信`、`附诗`、`追伸`、`补遗回信`、`异文`
- `term_notes` 用于挂接该段内出现的疑难词解释

---

## 3. 推荐数据结构

如果后端最终消费 JSON，推荐最少结构如下：

```json
{
  "work": "一人称",
  "edition": "bilingual_parallel",
  "groups": [
    {
      "group_id": "G01",
      "group_label": "第01组通信",
      "date_label": "3/10",
      "units": [
        {
          "unit_id": "G01-U01",
          "speaker": "我",
          "type": "来信",
          "source_refs": ["1-1", "1-2"],
          "paragraphs": [
            {
              "para_id": "G01-U01-P01",
              "source_md": "……",
              "target_md": "……",
              "notes": [],
              "term_notes": []
            }
          ]
        }
      ]
    }
  ]
}
```

如果前端已经有成熟结构，不必强行改成这一版。
但至少应保证上面的语义字段能一一映射。

---

## 4. 段落对齐规则

### 4.1 散文段落

- 原文一个自然段，原则上对应译文一个自然段
- 只有在中文必须拆句才能读通时，才允许一对二
- 如果发生一对二或二对一，必须保留显式 `para_id` 关联

### 4.2 诗歌段落

- 诗按“行”而不是按自然段对齐
- 一行原文原则上对应一行译文
- 不把整首诗并成一段散文式译文

### 4.3 块引用与页边补记

- `> ` 块独立成段，不并入邻近正文
- 同纸接写若在整理阅读版中已拆开，则继续独立对照

### 4.4 删除线

- `~~删除~~` 保留在 `source_md`
- `target_md` 同样保留删除线
- 不允许把删除内容偷偷并入正式译文

---

## 5. 注音与释义对接规则

## 5.1 不改动你现有的 `{|}` 体系

你已经有成熟的前后端渲染方案，因此这里的原则很明确：

- 不改成 HTML ruby
- 不改成脚注替代
- 不另发明第二套注音语法

后续整理和翻译时，只做与你现有 `{|}` 体系兼容的标注。

## 5.2 建议支持的三种语义层

为了兼顾“只有读音”和“读音加释义”两种需求，建议把 `{|}` 体系视为承载以下三类信息：

1. 原词
2. 读音
3. 释义

推荐语义模型如下：

- `原词`
- `原词 + 读音`
- `原词 + 读音 + 释义`

如果你的前端当前已经采用了固定格式，例如：

- `{詞|よみ}`
- `{詞|よみ|释义}`

那后续输出就直接遵守这一套，不再另改。

## 5.3 当前项目中的使用原则

这部作品里，注音和释义不应滥用。
默认只在以下几类位置加：

- 生僻或不直观的和语词
- 典故性专名
- 可能误读的汉字词
- 对理解情节或诗意推进有帮助的关键词
- 你已经在原稿中主动标出的疑难词

不建议把所有日文汉字词都加上读音。
那会把文学阅读变成语言教材。

### 5.4 原文层与译文层的关系

推荐做法：

- 注音与释义主要挂在原文层
- 译文层只在必要时保留解释性信息
- 如果某个词的双关或典故无法在译文中自然呈现，再在 `notes` 或译注中补
- 对于 `白妙`、`唐紅`、`晴る` 这类“光有读音仍不够”的词，必须额外挂到 `term_notes`

也就是说：

- `source_md` 负责保留词面、读音、原作质感
- `target_md` 负责给出中文文学化表达
- `notes` 负责处理无法内化进译文的信息
- `term_notes` 负责处理“这个词究竟是什么、在本作里为什么重要”

### 5.5 疑难词解释的来源

后续疑难词解释的基线文件为：

- [疑难词与文化注释表.md](/Users/kiharari/DEV/一人称小说/docs/annotations/疑难词与文化注释表.md)

建议前端消费时至少支持以下字段：

- `term`
- `reading`
- `short_gloss`
- `work_function`
- `long_note`

其中：

- `short_gloss` 用于悬浮短释义
- `work_function` 用于说明它在小说中的功能
- `long_note` 用于展开式长注

## 6. 翻译输出格式要求

从下一批翻译开始，建议不再只输出“译文正文”，而是输出“双语对照底稿”。

每个阅读单元应至少包含以下部分：

1. 单元元数据
2. 原文段落列表
3. 译文段落列表
4. 注音或释义标记
5. 译者备注

推荐 Markdown 交付形态如下：

```md
## 第01组通信｜3/10

### 单元 G01-U01｜我｜来信

#### P01
- 原文：……
- 译文：……

#### P02
- 原文：……
- 译文：……

[译者备注]
- `某词` 保留原文歧义
- `某典故` 待核对通行译名
```

如果后续要直接转 JSON，也可以先保留这一层中间稿，再由脚本转换。

## 8. 当前建议的工作顺序

现在最合理的推进顺序是：

1. 先按批次开始翻译
2. 翻译时直接产出“逐段对照底稿”
3. 遇到需要注音或释义的词，按你现有 `{|}` 规则写入原文层
4. 每个批次完成后，再做一次段落对齐检查
5. 最后再把整部作品统一导入你的前后端项目

---

## 9. 当前结论

翻译后的主版本现在已经可以明确为：

**逐段对照的双语版，而不是原文版和译文版彼此分离。**

注音和释义层也不需要另起系统，而是：

**继续兼容你现有的 `{|}` 标记体系，由我在翻译与整理时按约定输出。**
